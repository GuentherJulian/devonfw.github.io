Motivation and Problem

Currently when I download the devon distribution, I get tons of scripts in the top-level directory of the IDE installation:

  • console[.bat]

  • create-or-update-workspace[.bat]

  • ps-console[.bat]

  • s2-create[.bat]

  • s2-init[.bat]

  • update-all-workspaces[.bat]

  • variables[.bat]

  • variables-customized[.bat]

  • env.sh

  • initialize.sh

  • uninstallUI.sh

  • vscode-workspaces_vs[.bat]

  • devcon[.bat]

  • devon[.bat]

IMHO this is quite messy. A user new to devon will ask himself:

  • Which of these scripts will I open my Eclipse? Answer is: none of them!

  • What do I need to do to setup the IDE? Answer is: create-or-update-workspace or update-all-workspaces

  • If there is uninstallUI where is installUI? Answer is IMHO: initialize installs the "UI" (Angular npm tools)

  • What is s2 and what do these s2 scripts do? Answer is: I do not have a clue either. Maybe Amazon S2?

  • If I run variables Script or env.sh nothing really seems to happen (and my environment is not properly set on linux/mac). What is that for? Answer is: These are no scripts to be executed. They are to be sourced by other scripts.

  • Why is there workspaces and workspaces_vs? Do I need both?

These are just a few of tons of such questions. Further there are many awesome features like IDEenv that people do not even are aware of.

Conclusion

In order to improve the UX we want to redesign the devon-ide. When things are simpler also less quetions will appear what also makes our life eaiser (less support effort, more happy users, etc.).

Proposal

I would like to propose that we reduce the number of scripts (in the main directory and used by end-users) to an ultimate minimum. With that I mean one or two.

So what if there is just one script called devon:

  • if the IDE is not initialized yet, it will do so

    • perform all the current things

      • copy settings.xml

      • update workspaces

      • …​

    • perform additional things

      • create settings-security.xml if not exists with random master password

      • invoke specific setup scripts that can be added as "plugins" in a specific folder, e.g. to install npm modules, clone git repositories into the workspace

      • …​

  • if the IDE is already initialized, it will be updated (update-all-workspaces)

For special features such as create-or-update-workspace «my-workspace» I could also do devon update «my-workspace».

Initialize/update workspace(s)

I have already reworked the ide-configurator to be independent of Eclipse. Hence, we can use the same tool also for VS Code, IntelliJ, Android-Studio, or whatever your favorite IDE may be (We do not want to support a zoo of IDEs but could allow the community to provide support for "unofficial" IDEs. However, I see "VS Code" as something we should officially support besides Eclipse as it is evolving great and gives us free Angular support while Angular development with Eclipse is still a pain). The only aspects we need to know are:

  • What is/are the IDE(s) you want to use in your project?

  • Where are the according settings (workspace templates) located?

  • Where do we need to write the resulting configuration to?

Actually the answer to the first question implicitly also answers the other questions. How could we answer the first question?

  • Auto-detection from software folder: If Eclipse is present, we will setup Eclipse, if (also) VS Code is present, we will also setup VS Code. A challenge to this approach is that sometimes IDEs are installed globally and do not come with the software folder.

  • Auto-detection from settings folder: If there are templates for Eclipse, VS Code, IntelliJ, etc. then let us setup according workspaces.

  • Explicitly let the project (architect) configure this in a config-file (variables[.bat]?) shipped with the IDE archive.

JH: My current thought is that we can provide some magic for the simple default script but we should simply allow to call devon eclipse to start eclipse and devon vscode to run vscode in your current project and workspace. If the workspace is not initialized it will be done for you automatically.

Now how to launch my IDE?

Should we continue to create launch scripts like eclipse-«workspace»[.bat]?

IMHO most people do not actually use multiple workspaces. Maybe the easiest solution could be that the new devon script without any arguments is simply also launching the IDE. But we do not want to "accidentally" update the workspaces when the IDE is launched. I see the following options:

  • Do not worry. As long as people do not pull the settings from git nothing will change that is unintendet.

  • Do not update workspaces if called without arguments. Only if nothing is yet initialized, the script will to a setup. To update, you would explicitly call devon update. If no workspace is specified, all workspaces are updated.

WDYT? Put your thoughts here…​

JH: I already created a prototyp where devon eclipse can start eclipse, etc. You can also run advanced things such as devon eclipse ws-update run or devon eclipse ws-reverse (as replacement for update-eclipse-workspace-settings). My proposal for the non-geek user-script in the home directory (e.g. called ide[.bat] or start[.bat]) we should do all initializations, also update the workspace and then launch the IDE (if not already running, e.g. for eclipse we can figure out via lock file).

How about IDEenv and env.sh?

Many people might not even know what IDEenv and env.sh actually are. These are scripts that will find your devon-ide toplevel directory and initialize the environment variables (so you can run mvn, npm, etc. properly for your project). On linux and mac this has just recently turned into a tool that magically installs itself on the first usage and adds an "alias" to your shell(s) so you can always and anywhere just type IDEenv.

Questions:

  • Should this also be named devon?

  • Would this cause confusion due to namespace clashing? So ./devon would do something else than devon.

The vision that I have is that you can type devon [«options»] anywhere in your shell no matter in which directory you are:
